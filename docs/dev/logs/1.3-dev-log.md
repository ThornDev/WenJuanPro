## Design Decisions (Gap Resolutions)

### [GAP-1] multi `correct="3,1"` ordering policy — **Resolution: strict ascending required**

Per `docs/architecture/data-models.md` the domain type is `correctIndices: Set<Int>`, suggesting
order-agnostic semantics at the data layer. However `docs/stories/1.3-...-校验.md` §Dev Notes
explicitly specifies "多选为 `1,3` 逗号分隔**升序**集合". Dev chose the stricter reading: input
MUST be comma-separated, strictly ascending, no duplicates. Rationale:

1. Strict ascending catches author typos (`"3,1"` is far more likely an accidental descending
   listing than an intentional set; forcing ascending keeps config files diff-friendly).
2. If a future relaxation is wanted, it's a one-line change in `parseMultiChoice`; the reverse
   (tightening after accepting any order) is breaking.

Implementation: `parseMultiChoice` walks the parsed indices and rejects any `ints[k] <= ints[k-1]`
with `CONFIG_FIELD_INVALID(field="correct")`, message "多选正确答案非法：必须严格升序且不重复".
`UNIT-018` asserts this Failure path.

### [GAP-2] `readBytes` IOException policy in `ConfigRepositoryImpl` — **Resolution: catch per-file, emit Invalid**

Story 1.4 will render `ConfigLoadResult.Invalid.errors` verbatim in the bottom Sheet. Dev chose
to wrap IO failures per-file (rather than propagate to caller) so one corrupt file never prevents
other configs from loading. Implementation:

```kotlin
try {
    val bytes = fileSystem.readBytes(path)
    when (val pr = parser.parse(fileName, bytes)) { ... }
} catch (e: IOException) {
    Timber.w("readBytes failed for a config file; marking invalid")
    ConfigLoadResult.Invalid(
        fileName = fileName,
        errors = listOf(
            ParseError(line = 0, field = null, code = CONFIG_FIELD_INVALID,
                       message = "$fileName: 读取文件失败（IO 异常）"),
        ),
    )
}
```

Rationale: `ENCODING_INVALID` is reserved for "file is not UTF-8" (per AC1 Error Handling table),
so we use `CONFIG_FIELD_INVALID` with `line=0, field=null` to signal "file-level IO error". The
exception's message is intentionally NOT included in the user-visible text (log filter NFR9:
disk paths may contain sensitive info); Timber records the incident at warn level with no path.
`BLIND-ERROR-004` asserts this recovery path.

### [OkioFileSystem.listFiles] case-sensitivity — **Resolution: case-sensitive `.txt`**

`endsWith(suffix)` is case-sensitive on the JVM and on Android (ext4/F2FS are case-sensitive by
default). Rationale: research-grade tooling expects deterministic behavior; mixed-case `.TXT`
files from external sources are rare and easier for users to rename than to silently accept.
`INT-008` asserts only lowercase `.txt` files are returned.

### [Header line numbering] — missing header field uses best-known line

If the header entry itself is absent, the reported line is `1` (canonical top-of-file). If the
line was present but invalid (e.g. `configId` format fails `^[A-Za-z0-9_-]{1,64}$`), we use
the actual line where the `# configId: ...` comment appeared. This gives users accurate
"click-to-jump" context for invalid content while avoiding fictional line numbers for missing
content.

### [Section qid regex] `^\[Q[1-9]\d*\]$` — rejects `[Q0]`

Story UNIT-025 skeleton asserts `[Q]`, `[Q0]`, and `[Q1a]` all fail with `field="qid"`. Original
draft used `^\[Q\d+\]$` which would accept `[Q0]`. Tightened to `^\[Q[1-9]\d*\]$` to reject
leading-zero identifiers, matching the skeleton contract. `Q10`, `Q99`, `Q100` remain valid.
