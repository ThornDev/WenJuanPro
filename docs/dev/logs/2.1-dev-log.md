## Design Decisions (Gap Resolutions)

### [GAP-1] CameraX bind lifecycle & executor teardown — **Resolution: `DisposableEffect` + single-thread executor**

`CameraPreviewView` owns the `cameraExecutor` via `remember { Executors.newSingleThreadExecutor() }`
(BR-1.9) and the bound `ProcessCameraProvider`. `onDispose` unbinds + shuts down. We considered hoisting
the executor into the ViewModel, but that couples VM to Android executors; keeping it in the
Composable keeps the VM pure and aligned with `Dispatchers.Main`/`viewModelScope` idioms. The VM still owns
the _contract_ (`OnCameraBindFailed` intent + `PermissionDenied(CAMERA_BIND_FAILED)` state) — only the
hardware glue lives next to the view.

### [GAP-2] QR engine abstraction — **Resolution: `internal interface QrDecoder` seam**

The QA skeleton hinted at two options — Robolectric-backed `ImageProxy` or an injected `QrDecoder` — and we
chose the latter. `QrAnalyzer` ships with `QrDecoder.default() = ZxingQrDecoder` in production; tests pass
an anonymous `object : QrDecoder` stub. This keeps the unit tests pure JVM, avoids pulling Robolectric into
the scan module's test path, and lets UNIT-026 read the hints straight off the `internal object QrHints`
constant. The QA design document is still authoritative for scenario IDs; the implementation seam is an
internal refactor, not a contract change.

### [GAP-3] Snackbar ownership — **Resolution: VM-owned, UI side-effect-free**

`transientSnackbar` lives on every `ScanUiState` variant (`override val transientSnackbar: String?`).
Invalid-scan reduction writes the string; `SnackbarShown` intent clears it. Alternative: host state inside
the Composable via `remember { SnackbarHostState() }` and `LaunchedEffect` only. Rejected because the VM
already has the `content` and the i18n-formatted copy is derived from `context.getString(R.string
.scan_invalid_id_format, content)`, which we want inside the state machine so that `ScanViewModelTest`
assertions are meaningful. The screen still owns the physical `SnackbarHost`; it just reads the message from
state.

### [GAP-4] `Idle → NavigateBack` vs `Error → NavigateBack` for missing configId — **Resolution: keep `Idle`**

When `configId` is blank, the VM sends `NavigateBack` and intentionally leaves the state as `Idle`. We
considered surfacing a visible error state, but per BR-1.5 this is a developer-facing invariant
(enforced at ConfigList → Scan), not a user-facing outcome. UI flashes `LoadingBody` for one frame and then
pops; the user sees nothing. The `Timber.w` covers observability.

### [GAP-5] Lifecycle observer scope — **Resolution: `DisposableEffect(lifecycleOwner)`**

Matches Story 1.2's `PermissionScreen` precedent, so returning from system settings repopulates `Preview`
without a user-visible `重新申请权限` tap. Intentional: the ON_RESUME refresh rewrites `ScanUiState.Preview`
and clears any in-flight `transientSnackbar`; acceptable because a foreground/background round-trip is a
coarse user action and the last transient message is probably stale by then.

## Notes on Blind-Spot / Edge Behaviors

- **BR-1.10 (same-QR debounce)** — double-layered. `QrAnalyzer.fired: AtomicBoolean` drops further analyzer
  callbacks even if CameraX hands us two frames before the VM can `clearAnalyzer`; the VM then guards
  `onQrDecoded` by checking `currentState !is Preview`. Either gate alone would be sufficient; having both
  means a single stray frame can't slip through during the ~1ms state-transition window.
- **CAMERA_BIND_FAILED reuses `PermissionDenied`** — product direction is "same error chrome, different copy"
  per `Error Handling` table. `ScanScreen` inspects `state.reason` and swaps the title + hides the
  `OpenAppSettings` secondary button; the primary button's label becomes `重试`. Distinct `Error` variant was
  rejected as over-engineering.
- **BLIND-RESOURCE-002** — UNIT-027 proves `image.close()` still fires through a thrown
  `NotFoundException`. ZXing normally throws `NotFoundException` for every no-QR frame, which is
  swallowed inside `ZxingQrDecoder.decode` → `null` return; the analyzer-level `finally` handles any other
  surprise.
- **NFR9 log sanitation** — `ScanViewModel` logs `len=<int>`, `valid` / `reason=STUDENT_ID_INVALID`,
  `stateCls=<class name>`; **never** the QR content. `QrAnalyzer` logs only `cls=<exception class>` if
  decoding blows up. UNIT-030 asserts both of these as a regression gate.
- **i18n-safe invalid snackbar** — uses `R.string.scan_invalid_id_format` with `%1$s` placeholder rather
  than string concatenation; keeps the Chinese default copy readable and leaves room for future `values-en/`.
