### Infrastructure (T0)

**New:**

- `domain/model/ResultRecord.kt` — `ResultRecord(qid, type, mode, stemMs?, optionsMs, answer, correct, score, status)`
  + `enum class QuestionType { SINGLE, MULTI, MEMORY }` + `enum class ResultStatus { DONE, NOT_ANSWERED, PARTIAL, ERROR }`.
- `domain/model/Session.kt` — `Session(studentId, deviceId, config, sessionStart, resultFileName, cursor=0,
  completedQids=emptySet())` + companion `composeFileName(deviceId, studentId, configId, sessionStart)` per the
  `${deviceId}_${studentId}_${configId}_${yyyyMMdd-HHmmss}.txt` rule.
- `domain/model/AppFailure.kt` — `data class AppFailure(code, cause?)` (RuntimeException) +
  `class WriteFailedException(...)` + `class SsaidUnavailableException("SSAID_UNAVAILABLE")`.
- `data/result/InMemoryResultRepository.kt` — Story 2.5 前的中转实现：保留 `startSession` /
  `hasIncompleteResult`（沿用 NoopResultRepository 的语义，让 Story 2.2 Welcome 流程不回归）；新增 `openSession(session)` /
  `append(record): Result<Unit>` / `findResumable(...)`：内存 `MutableList<ResultRecord>` + `Mutex` 串行化追加，
  `findResumable` 暂返回 `null`（Story 4.1 接管）。
- `domain/usecase/ScoreSingleChoiceUseCase.kt` — pure function `(question, answer, stemMs, optionsMs) → ResultRecord`；
  1-based answer → 0-based scores 索引；`mode==ALL_IN_ONE` 时强制丢弃 `stemMs`（写 null）；越界/scores-options 长度不一致
  以 `IllegalStateException` 抛出供 FSM 包装为 `Errored` 态。
- `domain/usecase/StartSessionUseCase.kt` — 在 `@IoDispatcher` 上读 SSAID（null/blank → `Result.failure(
  SsaidUnavailableException)`），构造 `Session.composeFileName(...)`，调用 `ResultRepository.openSession(session)`，
  其它异常包装为 `AppFailure(code="SESSION_OPEN_FAILED")`。
- `domain/usecase/AppendResultUseCase.kt` — 在 IO 上调用 `ResultRepository.append(record)`，任何失败（包括
  `WriteFailedException` 与 generic `Throwable`）统一返回 `Result.failure(AppFailure(code="RESULT_WRITE_FAILED"))`。
- `domain/fsm/QuestionFsm.kt` — 表驱动 FSM；纯函数 `reduce(state, event, nowMs)`：
    - states: `Init / QuestionAllInOne / QuestionStagedStem / QuestionStagedOptions / Writing / WriteError /
      NextDecision / Terminated / Errored`
    - events: `Enter(question) / SelectOption(index) / OptionsSubmit(index) / OptionsTimeout / StemTimeout /
      WriteSuccess / WriteFailure / Retry / RetryExhausted`
    - `Writing.retriesLeft` 起始 3；`WriteFailure` 减一并 clamp 到 0；`RetryExhausted` 仅在 `WriteError(retriesLeft=0)`
      时转 `Terminated`。
    - 失序事件（如 `OptionsTimeout` 落在 `Writing` 上）一律返回原状态，把幂等守卫聚拢到 reducer 一层
      （满足 `BLIND-CONCURRENCY-001 / FLOW-002` 的"first-event-wins"语义）。
- `feature/question/{QuestionUiState,QuestionIntent,QuestionEffect}.kt` — sealed 接口三件套：
    - `UiState`: `Loading / SingleChoiceAllInOne / SingleChoiceStaged / RetryWriteBanner / Error`
      （所有变体都实现父接口的 `countdownProgress: Float` + `isWarning: Boolean` 属性）；
      `enum class Stage { STEM, OPTIONS }` 嵌套。
    - `Intent`: `OnEnter / SelectOption / Submit / TimerExpired / StageTransition / Retry`。
    - `Effect`: `NavigateNext(qid, studentId, configId) / NavigateComplete / ShowRetryBanner(retriesLeft) /
      NavigateTerminal`。
- `feature/question/QuestionViewModel.kt` — `@HiltViewModel`；从 `SavedStateHandle` 取 `studentId / configId`，
  依赖 `SessionStateHolder + ConfigRepository + StartSessionUseCase + AppendResultUseCase + QuestionFsm + Clock +
  @IoDispatcher`。Bootstrap 流程：`OnEnter → loadConfig（Holder cache 优先, 缺失再走 ConfigRepository.loadAll() 在
  IO 调度器） → setSelectedConfig → startSessionUseCase → openSession → enterCurrentQuestion(cursor=0)`。
  倒计时驱动：`viewModelScope.launch { while (isActive) { now = clock.nowMs(); if (elapsed≥duration)
  onIntent(TimerExpired)+break; renderUiFromFsm(now); delay(100) } }`，每 100 ms 重新计算 `countdownProgress` + 阈值
  `isWarning = remaining ≤ 5_000ms`。Submit / TimerExpired 均先 `cancelCountdown()` 再走 reducer，避免双触发。
  Persist 流程：FSM 出 Writing → 在 IO 上 `appendResultUseCase` → 成功 `WriteSuccess + advanceCursor + Enter(next)
  + NavigateNext` 或末题 `NavigateComplete`；失败 `WriteFailure`（reducer 自带 retries−1） → `retriesLeft=0` 触发
  `RetryExhausted + NavigateTerminal`，否则 `ShowRetryBanner(retriesLeft)` 等待用户 `Intent.Retry`。
  Intent 防抖：Submit 仅在 `selectedIndex != null` 接受；二次 Submit 在 `Writing` 状态下被 reducer 静默吞掉
  （BLIND-FLOW-002 / 003 / CONCURRENCY-001 全部走这条路径）。
