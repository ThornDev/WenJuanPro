# Dev Log: Story 2.5 — 结果文件实时落盘

## Agent Model Used
Claude Opus 4.6 (1M context)

## Implementation Summary

Replaced `InMemoryResultRepository` with `ResultRepositoryImpl` — a file-backed implementation that atomically persists each answer as a pipe-delimited line via `write → flush → fd.sync()`. Added serializers, file parser for resume detection, and diagnostic logger.

**T1: ResultFileWriter + ResultRepositoryImpl**

- `ResultRepositoryImpl.kt` — full `ResultRepository` implementation:

  - `append`: serialize → appendLine → Result.success / DiagLogger on failure
  - `findResumable`: scan directory, parse files, pick latest incomplete
  - `closeSession`: close writer, clear volatile reference
  - `resultsDir` internal var for testability

## Decisions & Notes

1. **WriteFailedException migration**: Moved from `domain.model` (RuntimeException) to `data.result` (IOException with `code` field). Kept backward-compatible default constructor for `AppendResultUseCase`.
2. **resultsDir testability**: Made `RESULTS_DIR` configurable via `internal var resultsDir` instead of modifying constructor signature (avoids DI graph change).
3. **DiagLogger made `open`**: Allows test subclassing without MockK (per project test strategy: JUnit4 + MockK, but simpler approach suffices here).
4. **InMemoryResultRepository preserved**: Left in `main/` source set; DI no longer binds it. Tests that use it can still import it directly.
