**As a** 学生（被试），
**I want** 在作答页按 config 指定的呈现方式（同屏模式 `all_in_one` 或分阶段模式 `staged`）完成一道单选题的阅读、选择与提交，并在倒计时归零前点击提交或静默超时后由系统自动记为未作答，
**so that** 研究员可以通过同一份 config 精确控制两种时序范式的作答窗口（同屏 = 自由分配读题与选择时间；分阶段 = 限定题干阅读阶段后独立进入选项阶段），从而比较两种呈现下被试的响应模式与得分，而学生端的交互、倒计时视觉阈值、阶段切换动效均完全由 config 驱动、无需学生感知模式差异（FR4 + FR8 + Story 2.5 结果写入）。

## Acceptance Criteria

### AC1: 同屏模式：题干 + 选项同屏，单倒计时

**Scenario**
```gherkin
GIVEN 当前题为 type=single, mode=all_in_one, durationMs=30000
WHEN 页面进入
THEN
  - 顶部进度条从 100% 起始按 30 秒倒计时线性递减
  - 题干区显示文字（或图片或混合）
  - 选项区显示 N 个选项（每行 2 或 3 个，由选项总数自动决定）
  - 学生点击某选项后该选项呈选中态，其他取消
  - 学生点击『提交』后立即记录答案、得分并写入结果文件
  - 倒计时归零仍未提交则记『未作答』并写入结果文件，自动进入下一题
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 选项渲染规则: 总数 ≤ 4 → 每行 2 个；5-9 → 每行 3 个；> 9 → 每行 3 个并垂直滚动 |
| BR-1.2 | 单选题选中是排他的；切换选项不重置倒计时 |
| BR-1.3 | 得分 = 该选项在 score 数组中的对应值 |
| BR-1.4 | 倒计时进度条颜色保持 Primary 蓝；剩余 ≤ 5 秒变橙；不弹任何提示 |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| 学生答案 answer | number | false | 1-N 之间的整数（N 为选项数），未作答时为空字符串 | （内部错误，不暴露给用户） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 结果写入失败 | `RESULT_WRITE_FAILED` | 数据保存失败，请联系研究员 | 阻塞下一题跳转，展示重试按钮（再次写入），重试 3 次仍失败则展示终止页 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 页面进入 | 倒计时立即开始；提交按钮置灰直到至少选中一个选项 |
| 选项点击 | 选中态动画（背景色淡入 100ms）；提交按钮可用 |
| 倒计时 ≤ 5 秒 | 进度条颜色变橙，无音效 |
| 提交或超时 | 立即跳转下一题，无确认弹窗 |

**Examples**
- **Input**: 30s 内选中选项 2 后提交
  **Expected**: 结果记录 `answer=2, score=scores[1], optionsMs=实际用时, status=DONE`
- **Input**: 30s 倒计时归零未提交
  **Expected**: 结果记录 `answer='', score=0, optionsMs=30000, status=NOT_ANSWERED`

### AC2: 分阶段模式：题干阶段独立倒计时，超时自动进入选项阶段

**Scenario**
```gherkin
GIVEN 当前题为 type=single, mode=staged, stemDurationMs=10000, optionsDurationMs=20000
WHEN 页面进入
THEN
  - 首先显示题干阶段：顶部进度条按 10 秒倒计时；屏幕仅显示题干，无选项
  - 题干阶段无任何作答按钮，学生点击屏幕无反应（但不报错）
  - 10 秒倒计时归零自动进入选项阶段（这是正常流转，不是异常）
  - 选项阶段显示选项 + 提交按钮 + 新的 20 秒倒计时进度条；题干已隐藏
  - 选项阶段提交或超时后写入结果，自动进入下一题
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-2.1 | 题干阶段无任何用户作答动作；题干阶段超时是『正常流转』，不记为错误也不影响后续选项阶段 |
| BR-2.2 | 题干阶段实际用时固定为 stemDurationMs（即倒计时全程）；记录到 stemMs 字段 |
| BR-2.3 | 选项阶段实际用时为学生进入选项阶段到提交/超时的耗时；记录到 optionsMs 字段 |
| BR-2.4 | 若学生在选项阶段未作答且超时，记 status=not_answered；得分为 0 |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 题干阶段任意异常（如系统中断 ANR） | `STAGE_TRANSITION_FAILED` | 数据保存失败，请联系研究员 | 整题作废，记录 status=error，写入结果后跳过到下一题 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 题干阶段点击 | 无视觉反馈，无错误提示（学生可能误操作，静默忽略） |
| 题干阶段倒计时归零 | 题干区淡出 200ms，选项区淡入 200ms，新进度条从 100% 起始 |
| 选项阶段提交或超时 | 立即写入结果并跳转下一题 |

**Examples**
- **Input**: 题干 10s + 选项 20s，学生在选项阶段第 8s 选中并提交
  **Expected**: 结果记录 `stemMs=10000, optionsMs=8000, answer=X, score=Y, status=DONE`
- **Input**: 题干 10s + 选项 20s，学生选项阶段未作答超时
  **Expected**: 结果记录 `stemMs=10000, optionsMs=20000, answer='', score=0, status=NOT_ANSWERED`

## Dev Notes

### Technical Constraints

- **时序精度约束（PRD NFR2）**：虽然本故事不涉及记忆题的 ±50ms 闪烁时序，但倒计时进度条递减与 `isWarning` 颜色过渡必须使用 `animate*AsState` / `LinearProgressIndicator` + `distinctUntilChanged` 驱动；**禁止** `Choreographer` / `withFrameNanos` / `Handler.postDelayed` [→ `docs/architecture/coding-standards.md#关键全栈规则`]
- **IO 线程隔离**：`ResultRepository.append` / `openSession` 必须在 `Dispatchers.IO` 执行；UI 状态更新切回 Main；VM 层用 `viewModelScope.launch { withContext(ioDispatcher) { … } }` 组合 [→ `docs/architecture/coding-standards.md#关键全栈规则`]
- **无直接 IO**：VM / Composable 严禁直接 `java.io.File` / `Okio`；所有文件操作走 `FileSystem` → `ResultRepository`；本故事的 `InMemoryResultRepository` 占位亦须封在 Data 层 [→ `docs/architecture/coding-standards.md#关键全栈规则`]
- **状态不可变**：`Question.SingleChoice` 数据类不可变；`QuestionUiState` 每次改变用 `update { current.copy(…) }`；禁止 `var` 暴露可变状态 [→ `docs/architecture/coding-standards.md#关键全栈规则`]
- **依赖白名单**：不得引入 OkHttp / Retrofit / Firebase / 任何网络库；本故事不需要新增任何依赖（Compose / Hilt / Coroutines 齐备）[→ `docs/architecture/tech-stack.md`]
- **分阶段模式 partial 不落盘**：staged 模式题干阶段不写文件；只有整题结束（选项阶段提交/超时/异常）才调用 `AppendResultUseCase` 一次，写一行 `ResultRecord`（`stemMs` 固定 = `stemDurationMs`）[→ `docs/prd/epic-2-scan-login-basic-questions.yaml` Story 2.5 BR-1.5；**本故事的内存 stub 遵循相同语义，Story 2.5 将此语义延续到真正文件写入**]
- **性能约束（PRD NFR3）**：题目切换（写入 → 下一题可见）≤ 300ms；选项点击视觉反馈 ≤ 100ms；`isWarning` 阈值触发到颜色切换 ≤ 200ms [→ `docs/front-end-spec.md#性能目标`]
- **日志内容过滤（PRD NFR9）**：Timber 日志仅打印 `qid / type / mode / status / retriesLeft`；**严禁** `Timber.d(answer)` 或 `Timber.d(score)`——答题明文不得出现在日志 [→ `docs/architecture/coding-standards.md#关键全栈规则`]
- **返回键消费**：`QuestionScreen` 必须 `BackHandler(enabled = true) { /* no-op */ }`；作答中途不允许物理返回键退回上一页（Story 4.1 断点续答之外的场景） [→ `docs/front-end-spec.md`]
- **字号遵循 Material 3**：题干 Title L 22sp / 选项 Body L 16sp / 顶栏"第 X / N 题" 14sp / 提交按钮 `Button` 默认 [→ `docs/front-end-spec.md#字体排版`]

- **JVM 单测先行（TDD）**：`QuestionFsmTest` / `ScoreSingleChoiceUseCaseTest` / `StartSessionUseCaseTest` / `AppendResultUseCaseTest` / `QuestionViewModelTest` 必须在对应生产代码前写好；每个测试独立、无共享状态

- **Compose UI 测试**：仅覆盖 1 条主路径（选项选中 + 提交触发 NavigateNext）+ 1 条分阶段切换路径 + 1 条 CountdownBar warning 态渲染；**不**对记忆题时序做自动化验证（本故事无 Memory 分支）
- **E2E 测试（`androidTest`）**：`QuestionFlowE2ETest` 用 `FakeResultRepository` 验证 `recordedResults` 列表内容；避免在 E2E 中断言真实文件系统（留给 Story 2.5）
- **Fake dispatchers**：`QuestionViewModelTest` 用 `StandardTestDispatcher` + `MainDispatcherRule`（参考 Story 2.1 `ScanViewModelTest` 同样模式）；倒计时使用 `TestScheduler` 推进虚拟时间验证 TimerExpired 分支
- **边界场景**：
  - `Question.SingleChoice.options.size == 1`（极端单选题——虽然业务上无意义，但不能崩）
  - `Question.SingleChoice.scores.size != options.size`（Parser 已拦截，但 VM 防御性处理：抛 IllegalStateException → Error 态）
  - staged 模式 `stemDurationMs == 0L`（立即切换选项阶段——不可能，但 VM 合法处理）
  - staged 模式 `options.size` 超过屏幕高度时选项栅格滚动可用（UI 测试验证）
  - 页面生命周期：`onStop` 期间倒计时 Flow 取消；`onStart` 恢复时按实际经过时间续算（**本故事 v1 先不实现生命周期续算，停留期间按照 Flow 暂停处理；Story 4.1 再细化**）
