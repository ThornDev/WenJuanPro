**As a** 研究员（config 作者），
**I want** App 能把 `/sdcard/WenJuanPro/config/` 下的 TXT 题库文件解析成可直接作答的不可变 `Config` 对象，且对任何格式/字段违规都给出**行号 + 字段 + 原因**三要素的错误信息，
**so that** 我可以独立验证自己编写的题库是否合法，不用把 App 当黑盒；合法的 config 进入测评集选择页，非法的 config 被明确标记为损坏并呈现完整错误列表。

## Acceptance Criteria

### AC1: 解析合法 config TXT 文件

**Scenario**
```gherkin
GIVEN /sdcard/WenJuanPro/config/ 下存在一份合法的 cog-mem-2026q3.txt
WHEN App 启动后扫描该目录并调用 ConfigRepository.loadAll()
THEN
  - 解析器返回 Config 对象，包含 configId、title、题目数组
  - 每道题包含 qid、type (single/multi/memory)、mode (all_in_one/staged)、durationMs / stemDurationMs+optionsDurationMs、题干、选项、正确答案、得分
  - 解析过程在 IO Dispatcher 执行，不阻塞主线程
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | config 文件 header 必须包含 configId 与 title 两个字段；缺失任一字段则该 config 标记为损坏 |
| BR-1.2 | configId 缺失时回退使用文件名（去 .txt 扩展）作为 configId（仅当 title 存在时生效；若 title 也缺失仍判损坏） |
| BR-1.3 | 题目以 `[Q1] [Q2] ...` 形式分节，节内 `key:value` 行；section 顺序即题目顺序 |
| BR-1.4 | 选项分隔符为竖线 `\|`；选项可包含图片引用 `img:文件名.png`（文件须存在于 `/sdcard/WenJuanPro/assets/`） |
| BR-1.5 | memory 题型固定 64 格点阵，10 个蓝点位置由 config 中的 `dotsPositions` 字段指定（10 个 0-63 索引） |
| BR-1.6 | 文件编码必须为 UTF-8（无 BOM）；BOM 存在时报警告（`ConfigWarning`）但仍可解析 |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| configId | string | true | 1-64 字符，仅限字母、数字、连字符、下划线 (`^[A-Za-z0-9_-]{1,64}$`) | configId 格式非法：只允许字母、数字、-、_ |
| 题目类型 type | string | true | 枚举: `single \| multi \| memory` | 未知题型：{值}（合法值: single/multi/memory） |
| 呈现模式 mode | string | true | 枚举: `all_in_one \| staged` | 未知呈现模式：{值}（合法值: all_in_one/staged） |
| 倒计时 durationMs / stemDurationMs / optionsDurationMs | number | true | 正整数，1000-600000（即 1 秒至 10 分钟） | 倒计时时长非法：必须为 1000-600000 毫秒之间的正整数 |
| memory 题 dotsPositions | array | true | 恰好 10 个互不重复的 0-63 整数 | 记忆题点位非法：必须为 10 个互不重复的 0-63 索引 |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| config 文件 header 缺失必填字段 | CONFIG_HEADER_MISSING | {文件名}: 第 {行号} 行缺少必填字段 {字段名} | 该 config 标记为 invalid，不阻塞其他合法 config 的加载 |
| 题目字段不合法（类型、模式、倒计时越界） | CONFIG_FIELD_INVALID | {文件名} [{Qid}] 第 {行号} 行: {具体原因} | 该 config 标记为 invalid，错误详情可在测评集选择页查看 |
| 选项中引用的图片文件不存在 | ASSET_NOT_FOUND | {文件名} [{Qid}]: 选项引用的图片 {路径} 不存在 | 该 config 标记为 invalid |
| 文件编码非 UTF-8 | ENCODING_INVALID | {文件名}: 文件编码非 UTF-8，无法解析 | 该 config 标记为 invalid |

**Examples**
- **Input**:
  ```
  # configId: cog-mem-2026q3
  # title: 认知记忆测评 v1
  [Q1]
  type: single
  mode: all_in_one
  durationMs: 30000
  stem: 你今天感觉如何？
  options: 很好|一般|较差|很差
  correct: 1
  score: 0|0|0|0
  ```
  **Expected**: `Config(configId="cog-mem-2026q3", title="认知记忆测评 v1", questions=[Question.SingleChoice(qid="Q1", mode=ALL_IN_ONE, optionsDurationMs=30000, stem=Text("你今天感觉如何？"), options=[Text("很好"), Text("一般"), Text("较差"), Text("很差")], correctIndex=1, scores=[0,0,0,0])])`
- **Input**: header 缺失 title
  **Expected**: `ParseResult.Failure(errors=[ParseError(line=2, field="title", code=CONFIG_HEADER_MISSING, message="cog-mem-2026q3.txt: 第 2 行缺少必填字段 title")])`

## Dev Notes

### Technical Constraints

- **纯 Kotlin + 可 JVM 单测**：`ConfigParser` 禁止依赖 Android framework（`android.*` / `Context` / `SharedPreferences`）；输入为 `ByteArray` + `sourceName: String`，输出为 `ParseResult` 密封类；所有解析逻辑可在 `app/src/test/` 的纯 JVM JUnit4 + MockK 环境下运行 [→ coding-standards.md#IO-仅在-Repository, testing-strategy.md]
- **IO 仅在 Repository 层**：`ConfigParser` 不直接读文件；由 `ConfigRepositoryImpl` 先通过 `FileSystem.readBytes(path)` 拿到 `ByteArray` 再传给 parser；asset 存在性检查通过注入的 `FileSystem.exists(assetPath)` 调用，不使用 `java.io.File` [→ coding-standards.md#IO-仅在-Repository]
- **协程与 Dispatcher**：`ConfigRepositoryImpl.loadAll()` 必须 `withContext(ioDispatcher)`；`@IoDispatcher` 由本故事新建的 `DispatchersModule` 提供；**禁止** `Dispatchers.IO` 硬编码于业务类内部 [→ coding-standards.md#协程作用域, tech-stack.md 协程]
- **不可变数据**：`Config` / `Question` 及其 sealed 分支一律 `data class` + `val`；`questions: List<Question>` 由 parser 一次性装配后返回，禁止在 repo/VM 中再修改 [→ coding-standards.md#不可变-Config-Question-Session]
- **UTF-8 无 BOM**：读取使用 `Charsets.UTF_8`；BOM（`EF BB BF`）存在时降级为 `ConfigWarning` 不阻塞；GBK/GB2312 等非 UTF-8 编码一律 `ENCODING_INVALID` [→ coding-standards.md#UTF-8-无-BOM, file-storage-schema.md#Config-文件-Schema]
- **错误三要素**：每条 `ParseError` 必须包含 `line`（1-based）/ `field`（字段名字符串，可为 null 仅在文件级错误如 `ENCODING_INVALID`）/ `message`（用户可读文案，含 `{文件名}` / `{Qid}` 等上下文）；Story 1.4 将直接读取 `Invalid.errors` 渲染底部 Sheet [→ front-end-spec.md 第 170 行 "行号 + 字段 + 原因"]
- **错误聚合（不短路）**：parser 遇到单条违规后**继续解析其余部分**，最后一次性返回 `Failure(errors=[...])`；例外：文件级错误（`ENCODING_INVALID`）可短路返回
- **日志内容过滤**：Timber 只记录 `sourceName` / `errorCount` / `questionCount` 等枚举与数字；**禁止** 打印题干、选项、正确答案等题目明文（NFR9） [→ coding-standards.md#日志内容过滤]
- **依赖白名单**：本故事仅需 Okio 3.9.0（已在 tech-stack.md 列出）；不引入任何额外 JSON / YAML 库（TXT 手工解析）；kotlinx.serialization 已在 1.1 引入可复用但本故事无需用到 [→ coding-standards.md#依赖白名单, tech-stack.md#文件-IO]
- **Hilt Scope**：`ConfigRepository` 与 `FileSystem` 使用 `@Singleton`；`ConfigParser` 无状态，可 `@Singleton` 或每次 `@Inject` 新建均可（选 `@Singleton` 减少对象分配） [→ coding-standards.md#Hilt-Scope]
- **Story 1.4 消费契约**：`Invalid.errors: List<ParseError>` 是 Story 1.4 "查看错误详情" Sheet 的数据源；`ParseError.line`/`field`/`message` 的命名与格式**一旦确定，Story 1.4 即依赖**；如需调整请发起 TCP。本故事对外公开契约为：`ConfigRepository.loadAll(): List<ConfigLoadResult>`、`ConfigLoadResult.Valid(Config)` / `Invalid(fileName, errors)`、`ParseError(line, field, code, message)`

    val configId: String,             // 1-64 chars, [A-Za-z0-9_-]+

    val parseWarnings: List<ConfigWarning> = emptyList(),

        get() = questions.sumOf { it.totalDurationMs }
}

sealed interface Question {
    val qid: String
    val mode: PresentMode
    val stemDurationMs: Long?
    val optionsDurationMs: Long
    val totalDurationMs: Long get() = (stemDurationMs ?: 0L) + optionsDurationMs

    data class SingleChoice(
        override val qid: String,
        override val mode: PresentMode,
        override val stemDurationMs: Long?,
        override val optionsDurationMs: Long,
        val stem: StemContent,
        val options: List<OptionContent>,
        val correctIndex: Int,           // 1-based
        val scores: List<Int>,
    ) : Question

    data class MultiChoice(
        override val qid: String,
        override val mode: PresentMode,
        override val stemDurationMs: Long?,
        override val optionsDurationMs: Long,
        val stem: StemContent,
        val options: List<OptionContent>,
        val correctIndices: Set<Int>,
        val scores: List<Int>,
    ) : Question

    data class Memory(
        override val qid: String,
        override val mode: PresentMode,
        override val stemDurationMs: Long?,
        override val optionsDurationMs: Long,
        val dotsPositions: List<Int>,    // 10 items, 0..63, distinct
        val flashDurationMs: Long = 1000L,
        val flashIntervalMs: Long = 500L,
    ) : Question
}

enum class PresentMode { ALL_IN_ONE, STAGED }

sealed interface OptionContent { data class Text(...); data class Image(...); data class Mixed(...); }

### Testing Requirements

- **JVM 单测（主战场，testing-strategy.md 明确指定）**：
  - `ConfigParserTest` 目标覆盖 ≥ 80% 行覆盖；必须覆盖 6 条 BR、5 条 Data Validation、4 个 ErrorCode 所有分支，以及错误聚合（同文件多错误不短路）与行号准确性两条核心不变式
  - `ConfigRepositoryImplTest` 必须使用 `kotlinx-coroutines-test` 的 `StandardTestDispatcher`，通过 `runTest { ... runCurrent() }` 验证 `withContext(ioDispatcher)` 切线程路径；假 FileSystem 桩返回可控文件集；覆盖空目录、全合法、全损坏、混合 4 种场景
- **Robolectric（可选）**：`OkioFileSystemTest` 可用 JUnit4 的 `TemporaryFolder` 规则在纯 JVM 中跑 Okio 真实 IO，不需 Robolectric
- **不做 UI 测试**：本故事无 UI
- **性能边界**：单 config 文件 ≤ 50 KB（5 题 × 平均 10 KB）解析耗时应 < 50 ms（tech-stack 未明确要求，但研究员期望刷新 ≤ 5s 扫描完；100 个文件算上 IO 应 < 1s）——非强制断言，但 Dev Log 记录实测
- **边界与易错点**（Dev 务必覆盖）：
  - `\r\n` vs `\n` 行尾混用
  - section 内出现空行或注释行 `# ...`（应跳过不计作有效 key:value，不影响行号）
  - `options: a||b` 连续分隔符（应算作 3 项：a, "", b）或判非法？→ **判非法**（空 option 无意义），代码中 `split("|")` 后过滤空串并按数量与 score/correct 对齐校验；有不一致则 `CONFIG_FIELD_INVALID`
  - 正确答案 `correct` 越界（单选 `correct=5` 但只有 4 个选项）→ `CONFIG_FIELD_INVALID`
  - `score` 长度与 options 长度不匹配 → `CONFIG_FIELD_INVALID`
  - 重复的 qid `[Q1]` 出现两次 → `CONFIG_FIELD_INVALID`（field="qid"）
  - 空文件 / 只有 header 无题目 → `CONFIG_FIELD_INVALID`（field=null, line=0 或 header 末行；"config 至少包含一道题"）
- **覆盖率目标**：Parser ≥ 80% 行覆盖，Repository 主路径 + 错误分支 ≥ 80%（testing-strategy.md §覆盖率目标）

### Implementation Summary

Implemented TXT config parser as a pure-Kotlin data pipeline: `FileSystem` abstraction +
`OkioFileSystem` implementation at the IO boundary; `ConfigParser` performs strict UTF-8 decoding
(BOM tolerated as warning, any other non-UTF-8 encoding returns `ENCODING_INVALID`), header
parsing with filename fallback for missing `configId`, `[Qn]` section splitting, per-type
dispatch to SingleChoice / MultiChoice / Memory parsers, and asset existence verification via
the injected `FileSystem`. All errors accumulate (no short-circuit) into a `List<ParseError>` and
are surfaced via `ParseResult.Failure`. `ConfigRepositoryImpl` wraps the pipeline behind
`withContext(@IoDispatcher)`, catches per-file `IOException` into `ConfigLoadResult.Invalid`, and
returns results stably sorted by `configId` (Valid) or `fileName` (Invalid). Full DI wiring via
new `DispatchersModule` + `ParserModule` (provides `okio.FileSystem.SYSTEM`) and extended
`DataModule` (binds `FileSystem` and `ConfigRepository`). 56 JVM test scenarios implemented;
two design gaps resolved in Dev Log (strict ascending for multi `correct`; per-file IO-exception
containment).
