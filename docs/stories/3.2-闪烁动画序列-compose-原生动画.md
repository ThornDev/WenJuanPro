**As a** 学生（被试），
**I want** 在记忆题进入后 500ms，看到 10 个蓝点按随机顺序依次闪烁（变黄 1000ms → 恢复 → 间隔 500ms → 下一个），
**so that** 我可以在心中记住蓝点的闪烁顺序，为随后的顺序复现点击做准备。

## Epic Context

**Epic**: 3 - 记忆闪烁点击题

实现 8x8 点阵渲染、Compose 原生动画驱动的闪烁序列、1.0x 半径命中判定、学生顺序复现的完整记忆题作答闭环；满足 ±50ms 视觉时序抖动目标。

**Story Deliverables**（本故事交付 Epic 3 的**闪烁动画层**，承接 Story 3.1 的点阵视觉基础，为 Story 3.3 复现阶段提供闪烁顺序数据）：

- `FlashSequenceGenerator`（`domain/usecase/FlashSequenceGenerator.kt`）：Fisher-Yates 洗牌算法生成本次闪烁顺序 `List<Int>`；每次进入记忆题重新洗牌
- `QuestionFsm` 扩展：新增 `FlashStart` / `FlashTick(index)` / `FlashComplete` 事件；`MemoryRendering` → `MemoryFlashing(currentIndex)` → `MemoryRecalling` 状态转换
- `QuestionViewModel` 扩展：`MemoryRendering` 进入后 500ms 启动闪烁协程；按 `flashDurationMs` / `flashIntervalMs` 节奏逐个更新 `dotStates[i]` 为 `FLASHING` → 恢复 `BLUE` → 下一个；全部完成后转入 `MemoryRecalling`
- `DotGrid` 扩展：`FLASHING` 状态已在 Story 3.1 预留占位色（黄色 #FFB300），本故事仅需通过 `dotStates` 列表驱动，无需修改 Canvas 绘制代码
- `MemoryQuestionContent` 扩展：闪烁阶段进度条保持 `isPaused = true` + `overlayText = "记忆中..."`（与 Rendering 阶段相同）

## Acceptance Criteria

### AC1: 10 蓝点按随机顺序依次闪烁一遍

**Scenario**
```gherkin
GIVEN 学生进入记忆题页面，10 蓝点已渲染（Story 3.1 完成）
WHEN 页面进入 0.5 秒后开始闪烁序列
THEN
  - App 在内存中生成本次闪烁顺序：dotsPositions 的随机排列（Fisher-Yates 洗牌）
  - 按生成顺序依次让对应蓝点『闪烁』：颜色变深（Primary 蓝 → 黄色 #FFB300）持续 flashDurationMs（默认 1000ms）
  - 相邻两次闪烁之间间隔 flashIntervalMs（默认 500ms），间隔期间所有点恢复初始 Primary 蓝
  - 10 个点全部闪烁一遍后进入复现阶段（Story 3.3 实现）
  - 闪烁阶段全屏禁用点击：学生任何点击都被静默忽略，不记录
  - 顶部进度条静止显示『记忆中...』文案（无倒计时）
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 闪烁动画必须使用 Compose 原生 API：通过修改 `dotStates` 列表中对应索引的 `DotState` 值（`BLUE` → `FLASHING` → `BLUE`）驱动 Canvas 重绘；**禁止** Choreographer / withFrameNanos 帧级手动调度 |
| BR-1.2 | 本次闪烁顺序须在每次进入记忆题时重新生成（Fisher-Yates 洗牌），不可跨题复用 |
| BR-1.3 | 闪烁顺序 `flashSequence: List<Int>` 保存在 `QuestionFsmState.MemoryFlashing` 中，供 Story 3.3 评分使用 |
| BR-1.4 | 实测闪烁时长抖动须 ≤ ±50ms（仅作为 NFR2 验收依据；不阻塞业务逻辑）|
| BR-1.5 | 闪烁阶段顶部进度条静止显示『记忆中...』文案（无倒计时）；进入复现阶段后才开始倒计时 |
| BR-1.6 | 闪烁阶段 `onTap` 保持 `null`（复用 Story 3.1 DotGrid 的禁用逻辑），学生点击无任何反应 |
| BR-1.7 | 闪烁序列总时长约：10 × flashDurationMs + 9 × flashIntervalMs = 10×1000 + 9×500 = 14500ms |
| BR-1.8 | 闪烁协程使用 `viewModelScope.launch` + `delay()` 驱动节奏；**不使用** `LaunchedEffect` 或 `Animatable`（因为状态由 ViewModel 管理而非 Composable） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 闪烁协程异常（极端低端机型 delay 被中断） | `ANIMATION_FAILED` | （用户不可见） | 记录 Timber.w 日志，标记本题 `status=ERROR`，`answer=""`，`score=0`，跳过到下一题 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 页面进入后 500ms | 首个蓝点开始闪烁（变黄 1000ms → 恢复蓝 → 间隔 500ms → 下一个） |
| 闪烁序列完成 | 全部点恢复初始蓝色，MemoryPhase 进入 Recalling（Story 3.3 接管） |
| 闪烁阶段任何点击 | 无视觉反馈，无声音，静默忽略 |

**Examples**
- **Input**: 10 蓝点 + 默认 flashDurationMs=1000ms / flashIntervalMs=500ms
  **Expected**: 总闪烁时长约 14500ms（10×1000 + 9×500），误差 ≤ ±50ms
- **Input**: flashSequence=[7, 3, 44, 19, 58, 12, 30, 22, 37, 51]（随机示例）
  **Expected**: 索引 7 先变黄 1s → 恢复蓝 → 0.5s → 索引 3 变黄 1s → ...

## Dev Notes

### Technical Constraints

- **Compose 动画约束（BR-1.1/BR-1.8）**：闪烁通过 `delay()` + 修改 `dotStates` 列表驱动 Canvas 重绘；**不使用** `Animatable`、`animate*AsState`（因为动画节奏由 ViewModel 协程控制，非 Composable 层）；**禁止** `Choreographer` / `withFrameNanos`
- **Fisher-Yates 洗牌**：使用 Kotlin stdlib `List.shuffled()`（内部是 Fisher-Yates）；每次进入记忆题重新生成
- **闪烁序列保存**：`flashSequence` 存储在 FSM 状态 `MemoryFlashing` 和 `MemoryRecalling` 中，供 Story 3.3 评分比对
- **协程生命周期**：闪烁协程使用 `viewModelScope.launch` 管理；页面销毁自动取消
- **日志安全（PRD NFR9）**：Timber 日志仅打印 `qid / type / phase / currentFlashIndex`；**严禁** `Timber.d(flashSequence)` 输出完整闪烁顺序
- **依赖白名单**：不新增任何依赖；`FlashSequenceGenerator` 为纯 Kotlin 类，通过 Hilt `@Inject` 注入
- **IO 线程隔离**：本故事为纯 UI + 内存操作，无 IO
- **性能目标**：每次 `dotStates` 更新触发 Canvas 重绘 < 8ms（单 Canvas 64 个 drawCircle，远低于 16ms 帧预算）
