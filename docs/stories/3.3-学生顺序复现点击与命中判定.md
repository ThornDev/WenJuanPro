**As a** 学生（被试），
**I want** 在闪烁序列结束后，按我记忆的顺序依次点击蓝点，系统实时判定命中并显示绿色选中态和序号，
**so that** 我可以完成记忆题的复现作答，系统根据我的点击顺序与闪烁顺序的前缀匹配数计分。

## Acceptance Criteria

### AC1: 学生按顺序点击 10 个点，命中判定使用 1.0× 视觉半径

**Scenario**
```gherkin
GIVEN 闪烁序列结束，进入复现阶段；当前题倒计时由 config optionsDurationMs 给定（如 60000ms）
WHEN 学生开始点击点阵
THEN
  - 顶部进度条显示倒计时（从满到空）
  - 学生每次点击系统判定命中：触点距离最近的蓝点中心 ≤ 该蓝点视觉半径（1.0×，不放大）
  - 未命中任何蓝点的点击不计入答案序列，不显示反馈
  - 命中蓝点后该点呈选中态（颜色变绿 #4CAF50 + 显示已点序号 1-10）
  - 已选中的蓝点不可重复点击（再次点击无反应）
  - 学生完成 10 个点击或倒计时归零时本题结束
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 命中半径 = 蓝点视觉半径 × 1.0（`cellSize * 0.35f`，不放大；研究场景精确性优先） |
| BR-1.2 | 命中判定使用『最近邻 + 距离阈值』算法：先找触点距离最近的蓝点，再判断距离 ≤ 半径 |
| BR-1.3 | 学生点击序列即 `answer`：例如点击顺序 `[12, 30, 7, ...]`，序列化为逗号分隔字符串 |
| BR-1.4 | 得分 = 点击序列与 `flashSequence`（expectedSequence）在前缀位置的匹配数。示例：expected=[3,7,12,...], answer=[3,7,19,...] → 前 2 位匹配 → score=2 |
| BR-1.5 | 倒计时归零时未点满 10 个，已点击的部分按 BR-1.4 计分；`status=DONE`（不算 NOT_ANSWERED，因学生有作答动作） |
| BR-1.6 | 学生完全未点击（answer 为空）则 `status=NOT_ANSWERED, score=0` |
| BR-1.7 | `DotGrid` 的 `onTap` 回调在复现阶段启用；接收点击像素坐标，由 `HitTestUtil` 判定命中 |
| BR-1.8 | `SELECTED` 状态：绿色 `#4CAF50` 实心圆 + 白色序号文字（1-10）居中 |
| BR-1.9 | 进度条文案从『记忆中...』变为无文案（或『请作答』），`isPaused = false`，实时倒计时 |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| answer | String | false | 逗号分隔的 0-63 索引序列 | （内部错误） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 命中算法返回多个候选（理论上不应发生） | `HIT_AMBIGUOUS` | （用户不可见） | 选择距离最小者；记录 Timber.w 日志 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 复现阶段进入 | 进度条文案变为倒计时；点阵接受点击 |
| 命中点击 | 点变绿 + 显示已点序号 |
| 未命中点击 | 无视觉反馈，静默 |
| 已点击的点再次被点 | 无反应 |
| 完成 10 个点击 或 倒计时归零 | 立即评分 + 写入结果 + 跳转下一题 |

**Examples**
- **Input**: expected=[3,7,12,19,22,30,37,44,51,58]，学生顺序点击 [3,7,12,19,22,30,37,44,51,58]
  **Expected**: score=10, status=DONE
- **Input**: expected=[3,7,12,...]，学生 60s 内仅点击 [3,7]
  **Expected**: score=2, answer="3,7", status=DONE
- **Input**: 学生从未点击直至超时
  **Expected**: score=0, answer="", status=NOT_ANSWERED

## Dev Notes

### Technical Constraints

- **命中判定算法（BR-1.1/BR-1.2）**：`HitTestUtil.hitTest()` 接收触点像素坐标 `(touchX, touchY)` 和 `cellSize`，遍历 `dotsPositions` 计算欧氏距离，找最近邻；判断 `distance ≤ cellSize * 0.35f`（半径 = cellSize × 0.7 / 2）；已选中的点跳过
- **前缀匹配评分（BR-1.4）**：逐位比对 `answer[i] == expected[i]`，首个不匹配即停止；score = 匹配数
- **DotGrid pointerInput**：`onTap` 非 null 时使用 `detectTapGestures { offset -> onTap(hitTest(offset.x, offset.y, ...)) }`；需传入 `cellSize` 供命中判定
- **SELECTED 序号绘制（BR-1.8）**：Canvas 内用 `drawContext.canvas.nativeCanvas.drawText()` 或 Compose `drawIntoCanvas` 绘制白色居中文字；字号 = `cellSize * 0.3f`
- **倒计时**：复现阶段复用 Story 2.3 的 `startCountdownForCurrentStage()` 模式
- **日志安全（PRD NFR9）**：Timber 日志仅打印 `qid / type / phase / selectedCount`；**严禁**输出完整 answer 序列
- **依赖白名单**：不新增任何依赖
- **IO 线程隔离**：评分为纯内存计算；结果写入复用 Story 2.5 的 `persistWriting()` 路径
