**As a** 学生（被试），
**I want** 把研究员发给我的学号二维码对准设备摄像头取景框，App 自动识别二维码内容并把它当作我的学号，识别成功后立即跳到欢迎确认页，
**so that** 我无需手动输入任何字符就能进入测评流程；同时研究员也无需担心我在键盘上敲错学号或在不同设备上写出不一致的学号——每次扫码都从同一张二维码读到完全相同的字符串，避免学号污染从而保证一份学号能精准匹配后续 result 文件命名与断点续答检索（FR3 + Story 4.1）。

## Acceptance Criteria

### AC1: 通过摄像头扫描二维码并将内容作为学号

**Scenario**
```gherkin
GIVEN 用户在测评集选择页选定一个有效 config 后进入扫码页
WHEN 用户将二维码对准取景框
THEN
  - App 调用 CameraX 预览 + ZXing Android Embedded 解码二维码内容
  - 识别成功后将二维码内容作为学号（studentId）存入会话状态
  - 自动导航到欢迎确认页（Story 2.2）
  - 全程不提供任何键盘手输入口
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 学号格式约束：1-64 字符，仅限字母、数字、连字符、下划线（正则 `^[A-Za-z0-9_-]{1,64}$`）；不符合则提示并继续扫码 |
| BR-1.2 | 扫码成功后摄像头立即停止（`imageAnalysis.clearAnalyzer()` + `cameraProvider.unbindAll()`），避免重复识别；同一会话不应重复入站第二次 `OnQrDecoded` |
| BR-1.3 | 需 `CAMERA` 运行时权限；首次进入时申请，拒绝则展示无摄像头提示并提供重新申请按钮 |
| BR-1.4 | 不支持手动输入学号（防止数据污染）；UI 不渲染任何 `TextField` / `BasicTextField` / 数字键盘组件；任何调试用模拟扫码入口（如 front-end-spec §页面 3 提到的 debug build 模拟扫码输入框）**仅在 `BuildConfig.DEBUG == true` 时启用**，Release 构建必须不可见且不可达 |
| BR-1.5 | 进入扫码页时 `SessionStateHolder.selectedConfigId` 必须非空（来自 Story 1.4 卡片点击落到 NavHost 的路由参数）；若解析路由参数后发现 `configId` 为空字符串或 null，**不**继续等待扫码，立即 `Effect.NavigateBack` 回测评集选择页并 Timber.w 记录（防御未来 NavHost 改动导致的脏数据） |
| BR-1.6 | 学号 `setStudentId()` 必须发生在 `Effect.NavigateToWelcome` 之前（同一 `viewModelScope.launch { }` 内顺序执行），保证 Story 2.2 进入时 `SessionStateHolder.studentId.value` 一定可读 |
| BR-1.7 | 屏幕方向：扫码页保持 portrait（与 MainActivity 全局 `screenOrientation="portrait"` 一致，本故事不需额外声明） |
| BR-1.8 | ZXing 解析仅启用 `BarcodeFormat.QR_CODE`（`MultiFormatReader.setHints(mapOf(DecodeHintType.POSSIBLE_FORMATS to listOf(BarcodeFormat.QR_CODE)))`），降低误识别（条形码 / DataMatrix 等不应被采纳） |
| BR-1.9 | `QrAnalyzer` 必须运行在 `Executors.newSingleThreadExecutor()` 上（CameraX `ImageAnalysis.setAnalyzer(executor, analyzer)` 第一参），不得占用主线程；`onDecoded` 回调切回 VM 时通过 `viewModelScope.launch(Dispatchers.Main)` 投递 Intent |
| BR-1.10 | 同一二维码连续帧识别去抖：`QrAnalyzer` 内部维护 `AtomicBoolean fired`，首次 `onDecoded` 命中后 `fired.set(true)`，后续帧立即返回不再投递（即便 VM 因协程调度延迟一两帧才处理 `clearAnalyzer`） |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| 二维码内容（学号 studentId） | string | true | 1-64 字符，正则 `^[A-Za-z0-9_-]+$` | 学号格式非法：仅允许字母、数字、-、_ |
| 路由参数 configId | string | true | 非空字符串；交由 Story 1.4 在卡片点击时保证；Story 2.1 仅做防御性校验 | （内部错误，Timber.w 记录后回退） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 学号格式非法 | STUDENT_ID_INVALID | 学号格式非法：{扫到的内容} | 保持扫码页打开，2 秒 `Snackbar` 后继续等待新二维码（`QrAnalyzer.fired.set(false)` 重置去抖标志，重新调用 `imageAnalysis.setAnalyzer(executor, analyzer)`） |
| CAMERA 权限被拒（首次或永久） | CAMERA_DENIED | 未获取摄像头权限，无法扫码登录 | 切换 `ScanUiState.PermissionDenied`；展示『重新申请权限』主按钮（再次拉起系统权限弹窗）+ 『前往系统设置』次按钮（`Intent.ACTION_APPLICATION_DETAILS_SETTINGS`） |
| 设备无后置摄像头 | CAMERA_UNAVAILABLE | 未检测到摄像头，无法扫码登录 | 切换 `ScanUiState.NoCamera`；展示返回按钮（`Effect.NavigateBack` 回测评集选择页）；不重试 |
| CameraX 绑定失败（如其他 App 占用摄像头） | CAMERA_BIND_FAILED | 摄像头被占用或初始化失败，请重试 | Timber.e 记录（仅 exception class + message）；切换 `ScanUiState.PermissionDenied`（复用同一错误态 UI），按钮文案改为『重试』 |
| 路由 `configId` 参数缺失 | CONFIG_MISSING | （内部错误，无用户文案） | Timber.w 记录；立即 `Effect.NavigateBack`，UI 无停留 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 进入扫码页（`OnEnter`） | 检查 `CameraPermissionRepository.isCameraGranted()`；未授权则发起运行时权限请求（`ActivityResultContracts.RequestPermission()`）；`hasBackCamera() == false` 时跳过权限请求直接进 `NoCamera` 态 |
| 权限授予完成 | 进入 `Preview` 态；`AndroidView` 工厂方法内通过 `ProcessCameraProvider.getInstance(context)` 异步绑定 `Preview` + `ImageAnalysis(setBackpressureStrategy=STRATEGY_KEEP_ONLY_LATEST, setOutputImageFormat=OUTPUT_IMAGE_FORMAT_YUV_420_888)` 到 `LifecycleOwner`；半透明黑色遮罩 `#000000 60% alpha` 中央挖空 70% 屏宽正方形（圆角 16dp，2dp 白色描边）|
| 识别成功（`OnQrDecoded(content)`） | （1）触觉反馈 50ms（`HapticFeedback.LongPress`，front-end-spec §动效 line 1098）；（2）`imageAnalysis.clearAnalyzer()` + `cameraProvider.unbindAll()` 立即停止预览；（3）调用 `StudentIdValidator.validate(content)`；合法 → `setStudentId` + `Effect.NavigateToWelcome(content, configId)`；非法 → 触发 `STUDENT_ID_INVALID` 处理 |
| 识别到非法学号 | 底部 `Snackbar('学号格式非法：xxx')` 显示 2 秒（`SnackbarDuration.Short`），摄像头继续工作（重新 `setAnalyzer`） |
| 点击左上角返回箭头 | `Effect.NavigateBack`（NavController `popBackStack` 回测评集选择页）；`onCleared` 中 `cameraProvider.unbindAll()` 防泄漏 |
| 进入错误态（`PermissionDenied` / `NoCamera` / `CAMERA_BIND_FAILED`） | 隐藏摄像头预览容器；居中渲染错误文案 + 操作按钮；不展示取景框 |

**Examples**
- **Input**: 扫码内容 `S001`
  **Expected**: 触觉反馈 50ms；`SessionStateHolder.studentId.value == "S001"`；`Effect.NavigateToWelcome("S001", "<当前 configId>")` 发送 1 次；NavController 跳转 `welcome?studentId=S001&configId=<id>`
- **Input**: 扫码内容 `含空格的 字符串`（含空格）
  **Expected**: `Snackbar('学号格式非法：含空格的 字符串')` 2 秒；摄像头继续；不写 `studentId`；不发 `NavigateToWelcome`
- **Input**: 扫码内容 65 字符长度的合法字符（超长边界）
  **Expected**: 视为非法，走 `STUDENT_ID_INVALID` 处理（正则上界 64）
- **Input**: 用户首次进入扫码页且系统权限弹窗点击「拒绝」
  **Expected**: `ScanUiState.PermissionDenied`；展示『重新申请权限』+『前往系统设置』；点击重新申请会再次拉起系统弹窗（Android 系统行为：连续两次拒绝后弹窗会变为不再询问，此时只能走系统设置入口）
- **Input**: 用户在系统设置中授权后回到 App
  **Expected**: `Activity.onResume` 触发 VM 重新检查权限；权限通过 → 自动从 `PermissionDenied` 切到 `Preview` 态，无需用户再次点击按钮
- **Input**: 设备只有前置摄像头（无后置）
  **Expected**: `hasBackCamera() == false`；`ScanUiState.NoCamera`；展示『未检测到摄像头』+ 返回按钮
- **Input**: 同一个二维码在扫码瞬间被识别 3 帧（高帧率）
  **Expected**: 仅触发一次 `OnQrDecoded`（去抖 `AtomicBoolean.fired`）；`Effect.NavigateToWelcome` 仅发送一次

## Dev Notes

### Technical Constraints

- **运行时权限版本边界**：`CAMERA` 自 Android 6.0 (API 23) 起需运行时申请，本 App `minSdk = 26`，无需兼容 API 23 以下；连续两次拒绝后系统弹窗会变为不再询问态（`shouldShowRequestPermissionRationale == false && permission != GRANTED`），此时『重新申请权限』再次拉起仍会被立即拒绝——产品决策：UI 仍提供该按钮（不做态判断隐藏），同时永远展示『前往系统设置』作为兜底
- **CameraX 库版本锁定**：`androidx.camera:camera-{core,camera2,lifecycle,view}:1.3.4`（与 tech-stack.md 一致）；不引入 `camera-extensions`（节省包体积，扫码场景用不上 HDR/夜景）
- **ZXing 引擎决策（v1.1 锁定）**：单押 `com.journeyapps:zxing-android-embedded:4.3.0`；**禁止引入 ML Kit Barcode Scanning**——会拖入 Google Play Services 动态模型下载，与"完全离线、零 Google 依赖"定位冲突（详见 `docs/architecture/tech-stack.md` 第 42 行的硬性决策）
- **包体积影响**：CameraX 五件套 + ZXing 预估 ≈ +1.2-1.8 MB（aar 内含 jniLibs，已经按 ABI 打包）；与 Release APK ≤ 12MB 上限（`security-performance.md`）有充足余量
- **冷启动影响**：CameraX 在 `ScanViewModel` 进入 `Preview` 态时才异步绑定，不在 `Application.onCreate` 中预热；遵守『App 冷启动 ≤ 2s』约束（NFR3）
- **NFR9 日志限制**：QrAnalyzer 与 ScanViewModel 中**禁止** `Timber.d(content)` 打印扫到的学号明文；只允许打印 `len` / `valid|invalid` / `reason` 等元数据
- **首次扫码到欢迎确认页耗时 SLA**：≤ 3 秒（front-end-spec §引言"学生使用效率"）；包括摄像头启动 + 识别 + 跳转；本故事仅承担"摄像头启动 + 识别 + 跳转"三段，欢迎页本身渲染由 Story 2.2
- **触觉反馈来源**：使用 Compose `LocalHapticFeedback.current.performHapticFeedback(HapticFeedbackType.LongPress)`，**不**直接调用 `Vibrator` 系统服务（避免增加 `VIBRATE` 权限）；`HapticFeedbackType.LongPress` 在主流 Android 设备上对应 ~50ms 短促振动，与 front-end-spec §动效 line 1098 描述一致

object StudentIdValidator {
    private val STUDENT_ID_REGEX = Regex("^[A-Za-z0-9_-]{1,64}$")

    fun validate(input: String): ValidationResult =
        if (STUDENT_ID_REGEX.matches(input)) ValidationResult.Valid
        else ValidationResult.Invalid("学号格式非法")

    sealed interface ValidationResult {
        data object Valid : ValidationResult
        data class Invalid(val reason: String) : ValidationResult
    }
}
