**As a** 研究员，
**I want** 学生每完成一道题后答案立即以原子追加方式写入设备本地 TXT 结果文件（含规范 header 元数据 + 管道分隔的数据行），且写入失败时由系统自动重试并在无法恢复时阻塞作答流程告知研究员，
**so that** 即使 App 崩溃或设备断电，已完成的题目结果均已落盘不丢失（flush + fsync 保证），且结果文件格式稳定可被后续数据分析脚本直接解析（FR5 + NFR4 + Story 4.x 续答依赖）。

## Epic Context

**Epic**: 2 - 扫码登录与基础题型作答

实现 QR 扫码 → 欢迎页 → 单选/多选题（同屏 + 分阶段模式）→ 结果实时落盘的完整主路径。完成本 Epic 后即可承载一份不含记忆题的离线问卷测评。

**Story Deliverables**（本故事交付 Epic 2 的**结果持久化引擎**，将 Story 2.3/2.4 的 `InMemoryResultRepository` stub 替换为真正的文件写入实现）：

- `ResultRepositoryImpl`（`data/result/ResultRepositoryImpl.kt`）：`ResultRepository` 接口的文件实现；注入 `FileSystem` + `DeviceIdProvider` + `Clock` + `@IoDispatcher`；实现 `openSession()` 创建结果文件并写入 header；实现 `append()` 原子追加数据行（整行组装 → write → flush → fsync）；实现 `findResumable()` 扫描结果目录匹配未完成会话
- `ResultFileWriter`（`data/result/ResultFileWriter.kt`）：封装单文件的低级写入操作；持有 `FileOutputStream` + `BufferedWriter`；提供 `writeHeader(session: Session, appVersion: String)` / `appendLine(line: String)` / `close()`；每次 `appendLine` 执行 write → flush → `fd.sync()`；异常统一转为 `WriteFailedException`
- `ResultRecordSerializer`（`data/result/ResultRecordSerializer.kt`）：纯函数，将 `ResultRecord` 序列化为管道分隔行字符串 `qid|type|mode|stemMs|optionsMs|answer|correct|score|status`；`stemMs` 为 `null` 时写 `"-"`；`type` / `mode` / `status` 写小写枚举名
- `SessionHeaderSerializer`（`data/result/SessionHeaderSerializer.kt`）：纯函数，将 `Session` + `appVersion` 序列化为 header 块（`key: value\n` × 5 + `---\n`）
- `FileSystem` 扩展：在 `core/io/FileSystem.kt` 接口新增 `openAppend(path: String): FileOutputStream` / `createFileIfNotExists(path: String)` 方法；`AndroidFileSystem` 实现对应的 `java.io.File` 操作
- `DiagLogger`（`data/diag/DiagLogger.kt`）：诊断日志写入器；在 `fsync` 失败或写入异常时将非敏感错误信息追加到 `/sdcard/WenJuanPro/.diag/app.log`；格式 `[ISO-8601] ERROR code: message`；**不记录 answer / score 等答题内容**
- `di/DataModule.kt` 更新：将 `InMemoryResultRepository` 绑定替换为 `ResultRepositoryImpl`（`@Singleton`）；新增 `DiagLogger` 绑定
- `BuildConfigProvider`（`core/BuildConfigProvider.kt`）：接口 `fun appVersion(): String`；实现读取 `BuildConfig.VERSION_NAME`；供 header 写入 `appVersion` 字段

## Acceptance Criteria

### AC1: 结果文件按规范命名并在首次写入时创建 header

**Scenario**
```gherkin
GIVEN 学生在欢迎页点击『开始答题』触发 ResultRepository.openSession(session)
WHEN 首次调用 append(record) 写入第一题结果
THEN
  - 若结果文件不存在则创建：/sdcard/WenJuanPro/results/{deviceId}_{studentId}_{configId}_{yyyyMMdd-HHmmss}.txt
  - deviceId = Settings.Secure.ANDROID_ID（已由 DeviceIdProvider 提供）
  - configId 取自 Config.configId
  - 时间戳取『开始答题』点击的瞬时时间（Session.sessionStart）
  - 首先写入 header 元数据（5 行 key: value + 1 行 ---）
  - 随后写入第一题数据行
  - header 和数据行均经 flush + fsync 落盘
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | header 格式（key: value 行，每行换行）：`deviceId: {val}` / `studentId: {val}` / `configId: {val}` / `sessionStart: {yyyyMMdd-HHmmss}` / `appVersion: {val}`；header 后紧跟 `---` 分隔行 |
| BR-1.2 | 数据行字段（管道分隔，9 字段）：`qid|type|mode|stemMs|optionsMs|answer|correct|score|status`；`stemMs` 仅 staged 模式有值，all_in_one 模式写 `-` |
| BR-1.3 | `type` 枚举值小写：`single` / `multi` / `memory` |
| BR-1.4 | `mode` 枚举值小写：`all_in_one` / `staged` |
| BR-1.5 | `status` 枚举值小写：`done` / `not_answered` / `partial` / `error` |
| BR-1.6 | 写入策略：先在内存组好完整行字符串，再一次性 write → flush → `FileDescriptor.sync()`；避免半行写入破坏文件 |
| BR-1.7 | 在 staged 模式下，题干阶段结束（自动进入选项阶段）时**不**写入任何内容；只有题目完整结束（选项阶段提交/超时/异常）才写一行完整 `ResultRecord`；这使续答判定简化为按 qid 是否存在 |
| BR-1.8 | 若 App 崩溃导致选项阶段未写入，下次续答时该 qid 视为未完成（整题重做） |
| BR-1.9 | 结果文件编码 UTF-8，无 BOM |
| BR-1.10 | `openSession()` 在 header 写入前自动 `mkdirs` 创建 `/sdcard/WenJuanPro/results/` 目录（若不存在） |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| deviceId | string | true | `Settings.Secure.ANDROID_ID` 返回值；非空 | 无法读取设备 ID |
| studentId | string | true | 非空，来自 QR 扫码 | （内部错误） |
| configId | string | true | 非空，来自 Config 解析 | （内部错误） |
| appVersion | string | true | `BuildConfig.VERSION_NAME`；非空 | （内部错误） |
| resultFileName | string | true | 匹配 `{deviceId}_{studentId}_{configId}_{yyyyMMdd-HHmmss}.txt` 格式 | （内部错误） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 结果目录不存在 | `RESULT_DIR_MISSING` | 结果目录不存在，已自动创建 | 自动 `mkdirs`，继续写入；Timber.w 记录 |
| 写入失败（磁盘满/权限丢失） | `RESULT_WRITE_FAILED` | 数据保存失败，请联系研究员 | 返回 `Result.failure(WriteFailedException)`；调用方（Story 2.3 VM）执行 <= 3 次重试 |
| fsync 失败 | `FSYNC_FAILED` | 数据落盘失败 | 写入 `.diag/app.log` 诊断日志（非敏感信息）；向上抛出 `RESULT_WRITE_FAILED` |
| header 写入失败 | `HEADER_WRITE_FAILED` | 数据保存失败，请联系研究员 | 删除已创建的空文件（避免残留空壳）；返回 `Result.failure` |
| 文件已存在（续答场景） | — | — | `openSession()` 检测到同名文件已存在时，打开已有文件并 seek 到末尾继续追加（不重写 header） |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| `openSession()` 成功 | 无 UI 反馈（静默创建文件，欢迎页已导航到首题） |
| `append()` 成功 | 无 UI 反馈（VM 收到 `Result.success` 后触发 `NavigateNext`） |
| `append()` 失败 | Story 2.3/2.4 VM 展示重试 Banner（本故事不控制 UI，仅返回 `Result.failure`） |

**Examples**
- **Input**: `deviceId=abc123, studentId=S001, configId=cog-mem-2026q3, sessionStart=2026-04-17T10:30:45`
  **Expected**: 文件 `/sdcard/WenJuanPro/results/abc123_S001_cog-mem-2026q3_20260417-103045.txt`，内容：
  ```
  deviceId: abc123
  studentId: S001
  configId: cog-mem-2026q3
  sessionStart: 20260417-103045
  appVersion: 0.1.0
  ---
  ```
- **Input**: Q1 同屏单选答完 — `answer=2, correct=1, score=0, optionsMs=24530`
  **Expected**: 追加行 `Q1|single|all_in_one|-|24530|2|1|0|done`
- **Input**: Q2 分阶段多选 — `stemMs=10000, optionsMs=8420, answer=1,3, correct=1,2, score=1, status=done`
  **Expected**: 追加行 `Q2|multi|staged|10000|8420|1,3|1,2|1|done`
- **Input**: Q3 同屏单选超时未作答
  **Expected**: 追加行 `Q3|single|all_in_one|-|30000||1|0|not_answered`

### AC2: 续答候选文件发现（findResumable）

**Scenario**
```gherkin
GIVEN 学生在欢迎页触发断点续答检测
WHEN 调用 ResultRepository.findResumable(studentId, configId)
THEN
  - 扫描 /sdcard/WenJuanPro/results/ 下匹配 {deviceId}_{studentId}_{configId}_*.txt 的文件
  - 读取文件数据行数（header 后的 --- 分隔行之后的行数）
  - 若数据行数 < config.questions.size → 视为未完成，返回 ResumeCandidate
  - 若有多个匹配文件，取 sessionStart 最新的一个
  - 若无匹配或所有匹配已完成 → 返回 null
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-2.1 | 文件匹配基于 `{deviceId}_{studentId}_{configId}_` 前缀 + `.txt` 后缀 |
| BR-2.2 | 已完成判定：数据行数（`---` 分隔行之后的非空行数）>= `config.questions.size` |
| BR-2.3 | 多个候选取 `sessionStart` 最新（按文件名时间戳排序） |
| BR-2.4 | `ResumeCandidate` 包含：`resultFileName` / `completedQids: Set<String>`（从数据行解析 qid 列）/ `cursor: Int`（= completedQids.size） |
| BR-2.5 | 文件解析出错（格式损坏）时跳过该文件，不阻塞新会话创建；Timber.w 记录 |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| ResumeCandidate.resultFileName | string | true | 匹配文件名格式 | （内部错误） |
| ResumeCandidate.completedQids | Set<String> | true | 从数据行首列解析；无重复 | （内部错误） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 结果目录不存在 | — | — | 返回 `null`（无候选文件） |
| 文件读取失败（权限丢失） | `RESULT_READ_FAILED` | — | 跳过该文件，Timber.w 记录，继续扫描 |
| 文件格式损坏（无 `---` 分隔行） | `RESULT_CORRUPT` | — | 跳过该文件，Timber.w 记录 |

**Examples**
- **Input**: 目录下有 `abc123_S001_cog-mem-2026q3_20260417-103045.txt`（含 3 行数据），config 共 5 题
  **Expected**: 返回 `ResumeCandidate(resultFileName=..., completedQids={Q1,Q2,Q3}, cursor=3)`
- **Input**: 目录下有同前缀两个文件（103045 和 110000），103045 含 5 行数据（已完成），110000 含 2 行
  **Expected**: 返回 110000 文件的 `ResumeCandidate`（103045 已完成，跳过）
- **Input**: 目录不存在
  **Expected**: 返回 `null`

## Dev Notes

### Technical Constraints

- **原子写入契约（BR-1.6）**：每次 `appendLine` 必须执行 `write(fullLine)` → `flush()` → `fd.sync()`；三步均在同一 IO 调度上下文中顺序执行；`fd.sync()` 确保数据从 OS 缓存刷到存储介质；**禁止** 用 `BufferedOutputStream.write` 分段写（可能产生半行）
- **IO 线程隔离**：`ResultRepositoryImpl` 所有公开方法须在 `Dispatchers.IO` 执行；调用方（`AppendResultUseCase`）已通过 `withContext(ioDispatcher)` 保证；Repository 内部**不再**额外切换 Dispatcher（避免双重切换开销）
- **无直接 IO 泄露**：VM / UseCase 不得持有 `FileOutputStream` / `File` 引用；所有文件操作封装在 `ResultFileWriter` → `ResultRepositoryImpl` → `FileSystem` 三层内
- **UTF-8 无 BOM**：`OutputStreamWriter(fos, Charsets.UTF_8)` 默认无 BOM；读取时若遇 BOM 则 Timber.w 记录但不阻塞
- **依赖白名单**：不新增任何第三方依赖；纯 `java.io` 操作
- **日志内容过滤（PRD NFR9）**：`DiagLogger` 仅记录 `code + message + timestamp`；**严禁**记录 `answer` / `score` / `studentId` 等答题或身份数据
- **`InMemoryResultRepository` 保留**：不删除 `InMemoryResultRepository`，将其降级为测试专用 Fake（移至 `test/` 目录或保留在 `data/result/` 但不在 `DataModule` 绑定）；JVM 单测和 UI 测试仍可使用内存实现
- **文件句柄生命周期**：`ResultFileWriter` 在 `openSession()` 时打开，在 `closeSession()` 或会话结束时关闭；避免长时间持有句柄导致 ANR 或 fd 泄漏；`@Singleton` Repository 需在 `onCleared`（App 退出）时确保关闭
- **并发安全**：单设备单学生串行作答，不存在并发写入场景；`ResultRepositoryImpl` 无需加锁，但 `@Singleton` 生命周期内 `resultFileWriter` 引用用 `@Volatile` 保护可见性
