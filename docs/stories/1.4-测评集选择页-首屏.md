**As a** 研究员（config 作者），
**I want** App 启动后直接呈现一个测评集选择页，把 `/sdcard/WenJuanPro/config/` 下的所有 TXT 题库渲染为可选卡片，并明确区分合法与损坏的 config（损坏的可点击查看行号级错误详情），
**so that** 我能在交付 App 给学生之前，一眼完成"config 有没有被 App 正确识别"的自检；同时，选中一份合法 config 后能无缝把控制权交给扫码登录页，让学生从此刻起进入线性作答流。

## Acceptance Criteria

### AC1: 展示所有 config 文件并区分有效/损坏

**Scenario**
```gherkin
GIVEN 已授权 + /sdcard/WenJuanPro/config/ 下有 N 个 config 文件（部分损坏）
WHEN App 进入测评集选择页
THEN
  - 页面顶部显示『请选择测评集』标题与刷新按钮
  - 以卡片列表呈现所有 config，每张卡片包含 configId、title、题数、状态徽标（有效/损坏）
  - 有效卡片可点击；损坏卡片置灰且不可点击，但可点击『查看错误详情』查看具体错误信息
  - 无任何 config 时展示空态：『请将 config TXT 文件放入 /sdcard/WenJuanPro/config/ 后点击刷新』
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 刷新按钮触发 `ConfigRepository.loadAll()` 重新扫描目录；扫描必须在 IO 线程，UI 显示加载指示器（最长 5 秒，超时显示失败提示） |
| BR-1.2 | 点击有效 config 卡片后，将该 configId 存入会话状态（`SessionStateHolder.selectConfig(configId)`）并导航到扫码登录页（Story 2.1） |
| BR-1.3 | config 列表按 configId 字典序排序（与 `ConfigRepositoryImpl.loadAll()` 已有的 `compareBy { configId or fileName }` 排序契约一致——损坏卡片以 `fileName` 参与排序） |
| BR-1.4 | 进入页面时自动触发一次 `loadAll()`；刷新按钮在加载期间置灰（`enabled = false`），加载结束后恢复 |
| BR-1.5 | 题数徽标：有效 config 显示 `{N} 题`；损坏 config 题数未知，显示 `? 题` |
| BR-1.6 | 状态徽标：有效=绿底白字 `有效`；损坏=红底白字 `损坏`（front-end-spec §ConfigCard 指定） |
| BR-1.7 | 损坏卡片整体 `Modifier.alpha(0.5f)` 且 `onClick = null`；仅右下角 `TextButton('查看错误详情')` 可点击，点击后弹出底部 Sheet |
| BR-1.8 | 错误 Sheet 使用 `ModalBottomSheet(skipPartiallyExpanded = true)`，最大高度 70% 屏高；内部 `LazyColumn` 渲染每条 `ParseError` 为 `第 {line} 行 · {field ?: "—"} · {message}`（等宽字体） |
| BR-1.9 | 整页**不**提供 `SwipeRefresh`（避免学生误操作）；刷新只能通过右上角按钮触发（front-end-spec §页面 2 交互说明） |
| BR-1.10 | 全部 config 损坏时**不**触发空态；展示全部损坏卡片 + 顶部横幅『当前无可用测评集』（front-end-spec §流程 1 边界情况） |
| BR-1.11 | 右上角"刷新按钮右侧的空白区域"支持长按 5 秒触发隐藏入口（`HiddenLongPressArea`），触发后发送 `ConfigListEffect.NavigateToDiagnostics`；该 Effect 当前**不**被 NavHost 消费（仅 Timber.d 记录），诊断页由后续 Epic 交付。长按期间**无任何视觉反馈**（front-end-spec §隐藏入口）|

**Data Validation**

N/A —— 本故事无新字段校验；消费 Story 1.3 已定义的 `Config` / `ParseError` 结构，字段约束已在 1.3 验证。

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| config 目录不存在 | CONFIG_DIR_MISSING | 未找到题库目录 /sdcard/WenJuanPro/config/，已自动创建 | `ConfigRepositoryImpl.ensureConfigDir()` 自动 `FileSystem.mkdirs(CONFIG_DIR)`；目录建好后再跑一次 `loadAll()`（仍为空时走空态） |
| 扫描超时（5s） | SCAN_TIMEOUT | 扫描题库超时，请检查目录是否存在大量文件 | 停止扫描（协程 `withTimeoutOrNull(5000)` 返回 null）；展示『重试』按钮；按钮点击再次触发 `Intent.Refresh` |
| 刷新过程中抛出未预期异常 | CONFIG_SCAN_FAILED | 扫描题库失败，请重试 | 捕获异常写入 Timber（仅 exception class + message，不含文件内容）；UI 切到 Error 态 + 重试按钮 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 进入页面（`init`） | 自动触发一次 `loadAll()`，展示居中 `CircularProgressIndicator`；5 秒超时后展示重试按钮 |
| 点击刷新按钮 | 发送 `Intent.Refresh`；按钮立即变灰（`enabled = false`）直到加载完成 |
| 点击有效 config 卡片 | 发送 `Intent.CardClicked(configId)`；VM 调用 `SessionStateHolder.selectConfig(configId)` → `Effect.NavigateToScan(configId)` |
| 点击损坏 config 的『查看错误详情』 | 发送 `Intent.ViewErrors(fileName)`；UI 弹出 `ModalBottomSheet` 展示 `List<ParseError>` |
| Sheet 关闭（点击关闭按钮 / 手势下拉） | 发送 `Intent.DismissSheet` |
| 右上角空白区域长按 ≥ 5s | 发送 `Intent.HiddenAreaLongPressed` → `Effect.NavigateToDiagnostics`；长按期间无任何视觉反馈；<5s 松手无效 |

**Examples**
- **Input**: 目录下有 `cog-mem-2026q3.txt`（合法）+ `broken.txt`（缺 title）
  **Expected**: 两张卡片：`cog-mem-2026q3` 标记『有效，5 题』可点；`broken` 标记『损坏』置灰，点击『查看错误详情』弹出 Sheet 显示 `第 2 行 · title · broken.txt: 第 2 行缺少必填字段 title`
- **Input**: 目录为空（但已存在）
  **Expected**: 空态文案 `请将 config TXT 文件放入 /sdcard/WenJuanPro/config/ 后点击刷新` + 次要按钮『刷新』
- **Input**: 目录不存在（首次启动授权后立刻进入）
  **Expected**: `ConfigRepositoryImpl.ensureConfigDir()` 自动 `mkdirs()`；扫描结果为空 → 展示空态（与目录存在但为空的视觉一致）
- **Input**: 目录下 100 个 config，扫描耗时 > 5 秒
  **Expected**: 加载指示器 5 秒后切换为『扫描题库超时，请检查目录是否存在大量文件』+ 重试按钮
- **Input**: 用户在右上角空白区域持续长按 5 秒
  **Expected**: 长按期间 App 界面**无任何视觉反馈**；5 秒到达时发送 `Effect.NavigateToDiagnostics`（Timber.d 记录，NavHost 未消费即不跳转）
- **Input**: 点击有效卡片 `cog-mem-2026q3`
  **Expected**: `SessionStateHolder.selectedConfigId = "cog-mem-2026q3"`；Effect 携带 `"cog-mem-2026q3"` 发送；NavHost 消费并 `navigate("scan?configId=cog-mem-2026q3")`（Story 2.1 将真正渲染扫码页；本故事只声明路由占位）

## Dev Notes

### Technical Constraints

- **纯 Compose + Material 3，无第三方 UI 库**：所有组件使用 Compose BOM 2024.09.02 + Material 3 已锁版本；**禁止**引入 Accompanist（`SwipeRefresh` 刻意不使用，见 BR-1.9）或其他三方 UI 库 [→ tech-stack.md L10-L12, coding-standards.md#依赖白名单]
- **MVI + StateFlow + Effect Channel**：`ViewModel` 持有 `MutableStateFlow<ConfigListUiState>` 与独立的 `MutableStateFlow<ErrorSheetState?>`（或合并，Dev 可选）；一次性事件（导航）走 `Channel<ConfigListEffect>(Channel.UNLIMITED).receiveAsFlow()`，与 `PermissionViewModel` 一致 [→ frontend-architecture.md §状态管理模式, coding-standards.md#Intent→Reducer-单向流]
- **5s 超时用 `withTimeoutOrNull(5_000)`**：**禁止** `delay` + 轮询；扫描协程本身必须 `withContext(ioDispatcher)`（由 `ConfigRepositoryImpl` 保证）；超时后不取消已 in-flight 的解析任务？——**决策**：由 `withTimeoutOrNull` 的取消语义自然取消子协程（`ConfigRepositoryImpl.loadAll()` 使用 `map { ... }` 是可取消的），Dev Log 记录实测超时恢复行为
- **Hilt Scope**：`SessionStateHolder` **必须** `@ActivityRetainedScoped`，绑定在 `ActivityRetainedComponent`；若误用 `@Singleton` 会导致"进程级单例"，与跨会话清理语义冲突（未来 Story 4 续答会重置 selectedConfigId）[→ coding-standards.md#Hilt-Scope, components.md L7]
- **IO 仅在 Repository**：VM / UseCase / UI **不得**直接调用 `java.io.File` / `Okio`；`ensureConfigDir` 扩展的 `FileSystem.mkdirs` 是唯一写入路径 [→ coding-standards.md#IO-仅在-Repository]
- **日志内容过滤**：Timber 仅记录 `configCount` / `invalidCount` / `fileName` / `errorCode` / 状态枚举；**禁止**打印 `title` / `stem` / 选项文案等题目明文（NFR9；即便是损坏 config 的错误 message 也不可直接进 Timber，只记录 `errorCount` + `ParseErrorCode`）[→ coding-standards.md#日志内容过滤]
- **错误文案走 strings.xml**：所有用户可见中文**必须**在 strings.xml 定义，禁止硬编码中文字面量在 Kotlin 源；错误码 `CONFIG_DIR_MISSING` / `SCAN_TIMEOUT` / `CONFIG_SCAN_FAILED` 仅存在于日志与内部状态枚举 [→ coding-standards.md#错误用户文案-vs-日志]
- **不可变 UiState**：`ConfigListUiState` 与 `ConfigCardUiModel` 均 `data class` + `val`；状态更新走 `_uiState.update { ... }`，禁止在 Composable 内部计算派生状态（如题数徽标）——应在 VM mapper 中预先 map [→ coding-standards.md#不可变-Config-Question-Session]
- **返回键策略**：测评集选择页是首屏，返回键=退出 App（NavHost 由 Navigation-Compose 默认 behavior 处理，无需 `BackHandler`）[→ front-end-spec.md L115]
- **隐藏入口零视觉反馈**：`HiddenLongPressArea` 不得使用 Ripple / Hover / 进度条等任何提示；5s 门槛使用 `SystemClock.elapsedRealtime()` 差值，**不得**用 `System.currentTimeMillis()`（墙钟可被用户篡改；`elapsedRealtime` 自启动单调递增） [→ front-end-spec.md L125-L128]
- **Story 2.1 消费契约（本故事对外 API）**：
  - `SessionStateHolder.selectedConfigId: StateFlow<String?>` — Story 2.1 扫码页将 `collectAsStateWithLifecycle()` 获取当前 configId
  - 导航路由参数：`scan?configId={configId}`（本故事新增占位路由；参数名 `configId` 锁定）
  - 二者**任一变更须发起 TCP**
- **Dev Log 必须记录**：
  - 5 秒超时实测样本（模拟 100 个 config 时的耗时中位数与 p95）
  - 隐藏长按时长门槛的真机验证（Android 12 / Android 14 各一次）
  - 损坏卡片 `alpha(0.5f)` 的可点击/不可点击语义是否被 TalkBack 识别（a11y 未来 Story 会改）

    when (result) {
        is ConfigLoadResult.Valid -> ConfigCardUiModel(
            configId = result.config.configId,
            title = result.config.title,
            questionCount = result.config.questions.size,
            isValid = true,
            errors = emptyList(),

            title = result.fileName,
            questionCount = null,

            errors = result.errors,
            sourceFileName = result.fileName,
        )
    }
