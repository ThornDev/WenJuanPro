**As a** 学生（被试），
**I want** 在作答页按 config 指定的呈现方式（同屏模式 `all_in_one` 或分阶段模式 `staged`）完成一道多选题的阅读、勾选与提交，多个选项可同时勾选/取消勾选，提交后得分按各选中选项 score 之和计算，
**so that** 研究员可以通过同一份 config 精确控制多选题在两种时序范式下的作答窗口，评估被试在多选场景中的决策模式与得分分布，而学生端的勾选交互、倒计时、阶段切换与单选题保持一致的视觉体验（FR4 + FR8 + Story 2.5 结果写入）。

## Epic Context

**Epic**: 2 - 扫码登录与基础题型作答

实现 QR 扫码 → 欢迎页 → 单选/多选题（同屏 + 分阶段模式）→ 结果实时落盘的完整主路径。完成本 Epic 后即可承载一份不含记忆题的离线问卷测评。

**Story Deliverables**（本故事交付 Epic 2 的**多选题作答引擎**，复用 Story 2.3 的 FSM / VM / Scaffold 骨架，差异点仅在选项勾选逻辑与评分函数）：

- `MultiChoiceAllInOneContent`（`ui/screens/question/MultiChoiceAllInOneContent.kt`）：同屏模式多选题子 Composable；复用 `CountdownBar` + 题干区 + 选项栅格（每行 2 或 3，由选项数自动决定）+ 底部『提交』按钮；与 `SingleChoiceAllInOneContent` 的差异：`OptionCard(isMulti = true)` 渲染 `CheckCircle` 图标；选中态为 toggle 而非排他
- `StagedMultiChoiceScaffold`（`ui/screens/question/StagedMultiChoiceScaffold.kt`）：分阶段模式 Scaffold；与 `StagedSingleChoiceScaffold` 结构一致（`AnimatedContent` 题干→选项淡出淡入切换）；选项阶段复用多选选项栅格
- `ScoreMultiChoiceUseCase`（`domain/usecase/ScoreMultiChoiceUseCase.kt`）：纯函数 `operator fun invoke(question: Question.MultiChoice, selectedIndices: Set<Int>?, stemMs: Long?, optionsMs: Long): ResultRecord`；得分 = `sum(question.scores[i - 1] for i in selectedIndices)`；未作答（`selectedIndices == null || empty`）→ `status = NOT_ANSWERED, score = 0`
- `QuestionFsm` 扩展：新增 `Enter(multi/all_in_one)` / `Enter(multi/staged)` reduce 分支；answer 从 `Int?` 扩展为 `Set<Int>`（toggle 语义：`ToggleOption(index)` 添加或移除）
- `QuestionUiState` 扩展：新增 `MultiChoiceAllInOne(…, selectedIndices: Set<Int>, …)` / `MultiChoiceStaged(…, selectedIndices: Set<Int>, …)` 子类型
- `QuestionIntent` 扩展：新增 `ToggleOption(index: Int)`（替代单选的 `SelectOption`，toggle 语义）
- `QuestionViewModel` 扩展：`when (question)` 分发新增 `Question.MultiChoice` 分支；`ToggleOption` 处理 toggle set 逻辑；`Submit` 调用 `ScoreMultiChoiceUseCase`
- `OptionCard` 扩展：`isMulti = true` 时渲染右上角 `Icon.CheckCircle`（Primary 蓝，24dp）

## Acceptance Criteria

### AC1: 同屏模式：多选题选项可勾选/取消，得分按选项独立计算

**Scenario**
```gherkin
GIVEN 当前题为 type=multi, mode=all_in_one, durationMs=30000
WHEN 页面进入
THEN
  - 顶部进度条从 100% 起始按 30 秒倒计时线性递减
  - 题干区显示文字（或图片或混合）
  - 选项区显示 N 个选项（每行 2 或 3 个，由选项总数自动决定）
  - 学生可点击多个选项进行勾选；再次点击取消勾选
  - 提交按钮置灰直到至少勾选一个选项
  - 提交时记录已勾选选项序号集合，得分 = 各选中选项 score 之和
  - 倒计时归零未提交则记 not_answered 并写入结果文件，自动进入下一题
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-1.1 | 多选题不要求学生选择正确数量；只要勾选 >= 1 项即可提交 |
| BR-1.2 | 正确答案集合（`correctIndices`）用于评分对照（写入结果文件 `correct` 字段），但不向学生展示对错 |
| BR-1.3 | 得分 = `sum(score[i] for i in selected)`，其中 `score[i]` 为 `Question.MultiChoice.scores` 中对应选项的分值（0-based 索引 = 选项序号 - 1） |
| BR-1.4 | 选项渲染规则与单选题一致：总数 <= 4 → 每行 2 个；5-9 → 每行 3 个；> 9 → 每行 3 个并垂直滚动 |
| BR-1.5 | `OptionCard(isMulti = true)` 渲染右上角 `CheckCircle` 图标（Primary 蓝 `#1976D2`，24dp），视觉上区分多选与单选 |
| BR-1.6 | 选中/取消勾选不重置倒计时 |
| BR-1.7 | 倒计时进度条颜色保持 Primary 蓝；剩余 <= 5 秒变橙 `#FB8C00`；不弹任何提示 |

**Data Validation**
| Field | Type | Required | Rules | Error Message |
|-------|------|----------|-------|---------------|
| 学生答案 answer | string | false | 逗号分隔的选项序号集合（1-based），如 `"1,3,4"`；未作答时为空字符串 `""` | （内部错误，不暴露给用户） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 结果写入失败 | `RESULT_WRITE_FAILED` | 数据保存失败，请联系研究员 | 同 Story 2.3：阻塞下一题跳转，展示重试按钮，重试 3 次仍失败则展示终止页 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 页面进入 | 倒计时立即开始；提交按钮置灰直到至少勾选一个选项 |
| 选项点击（未选中） | 勾选：背景色淡入 100ms + 2dp Primary 蓝边框 + 右上角 CheckCircle 出现；提交按钮可用（若为首次勾选） |
| 选项点击（已选中） | 取消勾选：背景色淡出 100ms + 边框消失 + CheckCircle 消失；若勾选数从 1 变 0，提交按钮置灰 |
| 倒计时 <= 5 秒 | 进度条颜色变橙，无音效 |
| 提交或超时 | 立即跳转下一题，无确认弹窗 |

**Examples**
- **Input**: 30s 内勾选选项 1 和 3 后提交
  **Expected**: 结果记录 `answer='1,3', score=scores[0]+scores[2], optionsMs=实际用时, status=DONE`
- **Input**: 30s 倒计时归零未勾选任何选项
  **Expected**: 结果记录 `answer='', score=0, optionsMs=30000, status=NOT_ANSWERED`
- **Input**: 勾选选项 1、2、4，然后取消选项 2，提交
  **Expected**: 结果记录 `answer='1,4', score=scores[0]+scores[3], status=DONE`

### AC2: 分阶段模式：题干阶段独立倒计时，超时自动进入选项阶段（多选）

**Scenario**
```gherkin
GIVEN 当前题为 type=multi, mode=staged, stemDurationMs=10000, optionsDurationMs=20000
WHEN 页面进入
THEN
  - 首先显示题干阶段：顶部进度条按 10 秒倒计时；屏幕仅显示题干，无选项
  - 题干阶段无任何作答按钮，学生点击屏幕无反应（但不报错）
  - 10 秒倒计时归零自动进入选项阶段（这是正常流转，不是异常）
  - 选项阶段显示多选选项 + 提交按钮 + 新的 20 秒倒计时进度条；题干已隐藏
  - 选项阶段可勾选/取消多个选项；提交或超时后写入结果，自动进入下一题
```

**Business Rules**
| ID | Rule |
|----|------|
| BR-2.1 | 题干阶段无任何用户作答动作；题干阶段超时是『正常流转』，不记为错误也不影响后续选项阶段 |
| BR-2.2 | 题干阶段实际用时固定为 `stemDurationMs`（即倒计时全程）；记录到 `stemMs` 字段 |
| BR-2.3 | 选项阶段实际用时为学生进入选项阶段到提交/超时的耗时；记录到 `optionsMs` 字段 |
| BR-2.4 | 若学生在选项阶段未勾选任何选项且超时，记 `status=NOT_ANSWERED`；得分为 0 |
| BR-2.5 | 分阶段模式的选项阶段行为与 AC1 一致（多选 toggle + CheckCircle + 提交启用条件） |

**Error Handling**
| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| 题干阶段任意异常（如系统中断 ANR） | `STAGE_TRANSITION_FAILED` | 数据保存失败，请联系研究员 | 整题作废，记录 status=error，写入结果后跳过到下一题 |

**UI Interaction**
| Trigger | Behavior |
|---------|----------|
| 题干阶段点击 | 无视觉反馈，无错误提示（学生可能误操作，静默忽略） |
| 题干阶段倒计时归零 | 题干区淡出 200ms，选项区淡入 200ms，新进度条从 100% 起始 |
| 选项阶段提交或超时 | 立即写入结果并跳转下一题 |

**Examples**
- **Input**: 题干 10s + 选项 20s，学生在选项阶段第 8s 勾选选项 1 和 3 并提交
  **Expected**: 结果记录 `stemMs=10000, optionsMs=8000, answer='1,3', score=scores[0]+scores[2], status=DONE`
- **Input**: 题干 10s + 选项 20s，学生选项阶段未勾选任何选项超时
  **Expected**: 结果记录 `stemMs=10000, optionsMs=20000, answer='', score=0, status=NOT_ANSWERED`

## Dev Notes

### Technical Constraints

- **复用 Story 2.3 骨架**：本故事**不新建** `QuestionViewModel` / `QuestionFsm` / `QuestionScreen`，而是在现有文件中扩展 `MultiChoice` 分支；保持单一 VM + 单一 FSM 架构，避免代码膨胀
- **answer 序列化格式**：多选答案 `answer` 字段为逗号分隔的 1-based 选项序号，**升序排列**（如 `"1,3,4"` 而非 `"4,1,3"`）；方便 Story 2.5 写入结果文件时格式稳定可读
- **IO 线程隔离**：同 Story 2.3，`AppendResultUseCase` 在 `Dispatchers.IO` 执行
- **状态不可变**：`selectedIndices: Set<Int>` 每次 toggle 产生新 Set（`current.selectedIndices + index` 或 `- index`）；禁止 `MutableSet`
- **依赖白名单**：不新增任何依赖（Compose / Hilt / Coroutines 齐备）
- **分阶段模式 partial 不落盘**：同 Story 2.3，staged 模式题干阶段不写文件；整题结束才写一行
- **性能约束（PRD NFR3）**：题目切换 <= 300ms；选项勾选视觉反馈 <= 100ms；`isWarning` 阈值触发到颜色切换 <= 200ms
- **日志内容过滤（PRD NFR9）**：Timber 日志仅打印 `qid / type / mode / status / retriesLeft`；**严禁** `Timber.d(answer)` 或 `Timber.d(score)`
- **返回键消费**：复用 Story 2.3 的 `BackHandler`，无需额外处理
